# Безопасность и эксплуатация

## Простыми словами

Этот модуль описывает, как SRCHR защищает данные и как изменения доходят до dev-сервера без раскрытия базы и секретов.

## Зачем нужен модуль

Публичный каталог должен быть открыт гостям, но это не означает открытый доступ ко всем строкам и операциям Supabase. Правила безопасности отделяют чтение опубликованного контента от личных данных и административных действий.

## RLS и Data API

- RLS включён на таблицах публичной схемы;
- `anon` читает только явно разрешённые опубликованные данные;
- `authenticated` получает дополнительные действия согласно владельцу и членству;
- служебные таблицы и `schema_migrations` не выдаются клиентским ролям;
- политики используют стабильные серверные проверки и не доверяют пользовательским metadata;
- старые и новые имена полей поддерживаются только там, где нужна совместимость.

Анонимные политики отделены от authenticated-политик. Это предотвращает ситуацию, когда публичный запрос вызывает закрытую функцию и каталог неожиданно становится недоступен гостю.

## Секреты и расширения

- `SUPABASE_SERVICE_ROLE_KEY` доступен только серверу;
- публичный клиент получает только URL и anon key;
- неиспользуемый `pg_graphql` отключён, REST/PostgREST остаётся основным Data API;
- функции с повышенными правами не должны становиться публичным API.

## Доставка

Обычный deploy выполняет `git pull`, установку зависимостей, build и перезапуск PM2. Миграции не запускаются в deploy workflow, потому что приложение не обязано владеть таблицами базы.

SQL-патчи применяются отдельно владельцем Postgres через специализированный workflow или вручную. Патчи должны быть повторно применимыми настолько, насколько это возможно, и проходить проверку advisor после применения.

## Сеть

Прямые внешние порты Postgres, pooler и внутренних сервисов Supabase закрываются firewall. Публичными остаются только необходимые HTTP/HTTPS endpoints через gateway. Пустые n8n и NocoDB пока сохранены без изменения по операционному решению.

## Ограничения

Ротация production-секретов и финальная проверка внешней поверхности выполняются отдельным этапом перед запуском. Любое новое публичное чтение требует явного решения по GRANT и RLS, а не только изменения интерфейса.
