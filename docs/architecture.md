# Архитектура SRCHR

База знаний хранится рядом с кодом, версионируется в Git и обновляется вместе с функциональностью. Интерфейс `/admin/knowledge` читает эти файлы, отдельной таблицы документации в Supabase нет.

## Техническая основа

- Next.js App Router, TypeScript и Server Actions;
- Supabase Auth, Postgres, PostgREST и Storage;
- shadcn/ui и продуктовые компоненты `components/srchr`;
- GitHub Actions для проверки, доставки приложения и отдельных SQL-патчей;
- PM2 запускает dev-окружение на сервере.

## Зоны приложения

- публичная зона — каталоги и опубликованный контент;
- dashboard — личные данные, создание объектов и действия авторизованного пользователя;
- admin — модерация, пользователи, аналитика, системные события и документация;
- API и Server Actions — единственная доверенная граница для изменений данных.

## Модель доступа

Гость может просматривать опубликованные каталоги подрядчиков, экспертов, медиа, мероприятий, задач и запросов стоимости. Контакты профилей, интерактивные действия и закрытые страницы требуют входа. Детали задач и запросов стоимости недоступны гостю.

Авторизованный пользователь действует лично как эксперт либо от имени организации, в которой у него есть подходящая роль. Административные полномочия проверяются на сервере и не зависят от видимости кнопки в интерфейсе.

## Данные и безопасность

- RLS включён для таблиц в публичной схеме;
- анонимные и авторизованные политики разделены там, где у них разные права;
- `service_role` используется только серверным кодом;
- служебные функции находятся вне публичного пользовательского контракта;
- SQL-патчи безопасности и модулей применяются отдельным owner-процессом;
- обычный deploy не запускает миграции и не блокируется правами владельца таблиц.

## Правило обновления

При изменении бизнес-логики обновляется документ затронутого модуля и его дата в `docs/modules/index.json`. Документы фиксируют действующее поведение, а не копируют большие фрагменты кода или историю коммитов.

## Структура

- `docs/modules/index.json` — компактный индекс, статусы и зависимости;
- `docs/modules/*.md` — актуальная логика отдельных модулей;
- `/admin/knowledge` — интерфейс чтения для администраторов;
- `/dev/ui` — рабочий каталог компонентов дизайн-системы.

## Статусы

- `active` — используется в продукте;
- `beta` — работает, но контракт ещё может измениться;
- `planned` — описан, но не реализован;
- `deprecated` — оставлен для совместимости.

## Шаблон модуля

Каждый документ начинается с продуктового объяснения: что делает модуль, зачем он нужен и какие возможности даёт. Далее документ фиксирует бизнес-правила, доступ, статусы, таблицы, маршруты, реализацию, связи и ограничения.
